# Earnings Transcript Ingestion & Processing Pipeline

This implementation plan is split into two phases. **Phase 1** details the immediate work to build the ingestion pipeline. **Phase 2** outlines the high-level architecture for the subsequent AI processing, which will be detailed and implemented later.

## Phase 1: Data Ingestion (Focus)

Read Parquet files generated by the Harvester and natively load them into a PostgreSQL `t_earnings_transcript_raw` table using DuckDB.

### Architecture
- **Kafka Event**: Re-use the existing `DataFileReadyEvent` (with `dataset = "earnings_transcript"`).
- **Idempotency**: The existing `DataFileLoadServiceImpl` handles idempotency by checking `t_load_log`.
- **Database Loader**: A new `ParquetFileLoadService` implementation will use `PostgresParquetFileLoader` to execute a high-speed DuckDB UPSERT.
- **State Management**: The raw transcript records will be inserted with a default status of `'PENDING'`. PostgreSQL's native TOAST system will automatically compress the large `transcript_text` strings in the background.

### Proposed Changes

#### Database Module
**[NEW] src/main/resources/db/migration/V17__t_earnings_transcript_raw.sql**
Creates the `t_earnings_transcript_raw` table, partitioned yearly to allow easy dropping of old data.
```sql
CREATE TABLE t_earnings_transcript_raw (
    id VARCHAR(64) NOT NULL,
    ticker VARCHAR(10) NOT NULL,
    quarter VARCHAR(10) NOT NULL,
    year INTEGER NOT NULL,
    transcript_text TEXT NOT NULL,
    status VARCHAR(20) DEFAULT 'PENDING',
    extracted_at TIMESTAMPTZ NOT NULL,
    created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id, extracted_at)
) PARTITION BY RANGE (extracted_at);

CREATE INDEX ix_earnings_transcript_raw_ticker_quarter ON t_earnings_transcript_raw(ticker, quarter, year);
```

**[MODIFY] src/main/resources/db/migration/R__00_partition_management.sql**
Add `t_earnings_transcript_raw` to the dynamic yearly partition creation function:
```sql
SELECT create_yearly_partitions('t_earnings_transcript_raw', (NOW() - INTERVAL '5 years')::DATE, (NOW() + INTERVAL '2 years')::DATE);
```

#### Load Module
**[NEW] src/main/java/com/oraculum/load/service/impl/EarningsTranscriptFileLoadServiceImpl.java**
- `@Component("earnings_transcript")` implementing `ParquetFileLoadService`.
- Defines the `TARGET_TABLE_NAME = "t_earnings_transcript_raw"`.
- Uses `BULK_UPSERT_SQL` to insert the DuckDB staging data into `t_earnings_transcript_raw` while setting `status = 'PENDING'`.

## Phase 2: LLM Extraction (High-Level Overview)

Once the raw data is safely stored, decouple the slow LLM processing from the fast ingestion pipeline. *(Details to be finalized later)*.

### Conceptual Architecture
1. **Event Trigger**: After a successful load in Phase 1, publish an internal Spring Application Event (e.g., `RawTranscriptLoadedEvent`).
2. **Asynchronous Listener**: The `analyst` module listens to this event asynchronously via `@Async @EventListener`.
3. **LLM Processing**: Fetches the `'PENDING'` transcript and orchestrates an LLM batch process using Resilience4j circuit breakers.
4. **Final Storage**: Upserts into a final "Gold" `t_earnings_transcript` table, and updates the raw table's status to `'COMPLETED'`.
